# 智能文件整理助手 - 产品路线图

## 产品愿景
打造一款**高效、资源占用低、纯本地隐私保护**的智能文件整理工具，覆盖软件/图片/文档三大场景。

---

## 版本规划

### V1.0 - 基础盘（当前阶段）
**目标**：稳扎稳打实现"分块哈希漏斗模型"，确立"性能极高"的第一印象

**核心功能**：
- ✅ 三层扫描流水线：元数据过滤 → 分块哈希 → 全量哈希
- ✅ 二进制精确去重（针对软件、压缩包）
- ✅ 基于 SQLite 的增量索引机制
- ✅ 生产者-消费者模型解耦 I/O 与 CPU 密集任务
- ✅ Worker 线程池并发处理

**技术栈**：
- SQLite 嵌入式数据库
- Worker Threads 多线程
- Bloom Filter 快速去重
- Memory-mapped I/O 加速

**性能目标**：
- 扫描 10 万文件 < 30 秒
- 内存占用 < 100MB
- 复扫效率提升 95%

## 架构设计

### 存储层（Single-File Infrastructure）
```
data/
├── file_index.db          # SQLite 主数据库
│   ├── files             # 文件元数据表
│   ├── hashes            # 哈希值表
│   ├── duplicates        # 重复文件组表
│   └── vectors           # AI 向量特征表 (sqlite-vss)
├── cache/
│   └── thumbnails        # 缩略图缓存
└── models/               # AI 模型文件
    ├── bge-micro-v2.onnx
    ├── clip-vit-b32.onnx
    └── qwen2.5-1.5b.gguf
```

### 任务调度层（Background Task Scheduling）
```
主线程 (UI)
    ↓ 提交任务
任务队列 (Priority Queue)
    ↓ 调度
Worker 线程池 (CPU Throttling)
    ↓ 执行
AI 推理 / 哈希计算 / 文件操作
```

**任务优先级**：
1. P0: UI 响应（立即执行）
2. P1: 传统哈希计算（并发执行）
3. P2: AI 特征提取（后台队列，可暂停）
4. P3: 模型加载/卸载（空闲时执行）

**资源节流策略**：
- 监控 CPU 使用率 > 80% → 暂停 P2/P3 任务
- 系统空闲 > 5 秒 → 恢复后台任务
- 内存占用 > 阈值 → 触发 LRU 模型卸载

### UI/UX 层（Design for Latency）
**异步渲染策略**：
1. **即时响应**：传统哈希结果立即显示
2. **骨架屏占位**：AI 计算中的项目显示脉冲动画
3. **渐进式加载**：AI 结果逐个显示，避免白屏
4. **透明度仪表盘**：右下角显示实时状态

**状态指示器**：
```
┌─────────────────────────────┐
│ 🚀 扫描完成: 10,000 文件     │
│ ⚡ 精确重复: 50 组 (立即显示) │
│ 🤖 AI 分析中: 45/1000...    │
│    内存占用: 320MB | CPU: 45%│
└─────────────────────────────┘
```

## 避坑指南

### ❌ 不要做的事
1. **不要在 MVP 引入 LLM** - 资源占用高，调试复杂
2. **不要要求独立数据库** - 保持单文件便携性
3. **不要在主线程跑 AI** - 会导致界面卡死
4. **不要一次性加载所有模型** - 按需懒加载

### ✅ 必须做的事
1. **渐进式交付** - 先快后准，先简单后智能
2. **数据局部性** - 元数据和向量存在同一个 SQLite
3. **后台调度** - 所有 AI 任务走队列，支持节流
4. **感知性能** - 用设计弥补算力不足
